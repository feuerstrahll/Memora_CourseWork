СЛАЙД 2: ЦЕЛЬ РАБОТЫ
================================================================================

ЦЕЛЬ РАБОТЫ

Разработка полнофункционального веб-приложения для управления архивными 
фондами с системой аутентификации, разграничением прав доступа и расширенными 
возможностями поиска и работы с документами.

Основные направления:
• Автоматизация процессов управления архивными документами
• Обеспечение безопасного доступа к архивным материалам
• Создание удобного интерфейса для различных категорий пользователей
• Реализация системы заявок на просмотр и оцифровку документов

СЛАЙД 3: ЗАДАЧИ РАБОТЫ
================================================================================

ЗАДАЧИ РАБОТЫ

1. Проектирование и реализация базы данных для хранения информации об 
   архивных фондах, описях и единицах хранения

2. Разработка серверной части приложения (Backend) с использованием NestJS:
   • Реализация REST API
   • Система аутентификации на основе JWT токенов
   • Ролевая модель доступа (RBAC)
   • Валидация входных данных

3. Разработка клиентской части приложения (Frontend) с использованием React:
   • Создание интуитивно понятного пользовательского интерфейса
   • Реализация расширенного поиска и фильтрации
   • Управление состоянием приложения
   • Визуализация статистики

4. Реализация функционала загрузки и скачивания файлов документов

5. Настройка инфраструктуры развертывания с использованием Docker

СЛАЙД 4: ПРЕДМЕТНАЯ/ОБЪЕКТНАЯ ОБЛАСТЬ
================================================================================

ПРЕДМЕТНАЯ/ОБЪЕКТНАЯ ОБЛАСТЬ

Предметная область: Архивное дело, управление архивными фондами

Объекты предметной области:

1. ФОНДЫ (Fond)
   • Код фонда
   • Название фонда
   • Описание
   • Хронологические рамки

2. ОПИСИ (Inventory)
   • Номер описи
   • Название описи
   • Принадлежность к фонду

3. ЕДИНИЦЫ ХРАНЕНИЯ (Record)
   • Шифр единицы хранения
   • Название
   • Аннотация
   • Даты документов
   • Объем
   • Уровень доступа
   • Прикрепленные файлы (PDF, DOCX)

4. КЛЮЧЕВЫЕ СЛОВА (Keyword)
   • Для категоризации и поиска документов

5. ПОЛЬЗОВАТЕЛИ (User)
   • Администратор
   • Архивариус
   • Исследователь

6. ЗАЯВКИ (Request)
   • На просмотр документов
   • На оцифровку документов

================================================================================

АЛГОРИТМ ШИФРОВАНИЯ

Для защиты паролей пользователей используется алгоритм BCRYPT:

• Алгоритм: bcrypt (Blowfish-based password hashing)
• Раунды хеширования: 10 (рекомендуемое значение для баланса безопасности и производительности)
• Реализация: библиотека bcrypt версии 5.1.1 для Node.js

Процесс:
1. При регистрации: пароль хешируется с помощью bcrypt.hash(password, 10)
2. При входе: пароль проверяется через bcrypt.compare(password, hash)
3. Пароли никогда не хранятся в открытом виде в базе данных

Преимущества bcrypt:
• Адаптивное хеширование (замедление при увеличении вычислительной мощности)
• Встроенная соль (salt) для каждого хеша
• Защита от атак перебора (brute-force)

ПРИМЕР ПОЛНОГО ЦИКЛА ВЗАИМОДЕЙСТВИЯ:

1. ПОЛЬЗОВАТЕЛЬСКОЕ ДЕЙСТВИЕ
   Пользователь нажимает кнопку "Создать запись"
   ↓
2. REACT COMPONENT
   Вызывает функцию handleSubmit()
   ↓
3. REACT QUERY MUTATION
   Вызывает recordsApi.create(data)
   ↓
4. AXIOS REQUEST
   POST /api/records
   Headers: { Authorization: "Bearer <token>", Content-Type: "application/json" }
   Body: { inventoryId: 1, refCode: "Д-123", title: "...", keywordIds: [1,2] }
   ↓
5. СЕТЕВОЙ УРОВЕНЬ
   TCP соединение, HTTP запрос на порт 3000
   ↓
6. NESTJS MIDDLEWARE
   CORS проверка → JWT Guard → Roles Guard → Validation Pipe
   ↓
7. CONTROLLER
   RecordsController.create() получает CreateRecordDto
   ↓
8. SERVICE
   RecordsService.create() валидирует данные, применяет бизнес-логику
   ↓
9. TYPEORM
   Преобразует объект в SQL: INSERT INTO records ...
   ↓
10. POSTGRESQL
    Выполняет SQL запрос, возвращает созданную запись
    ↓
11. ОБРАТНЫЙ ПУТЬ
    PostgreSQL → TypeORM → Service → Controller → HTTP Response
    ↓
12. AXIOS RESPONSE
    Получает { id: 123, refCode: "Д-123", ... }
    ↓
13. REACT QUERY
    Обновляет кеш, инвалидирует связанные запросы
    ↓
14. REACT COMPONENT
    Обновляет UI, показывает уведомление об успехе
    ↓
15. ПОЛЬЗОВАТЕЛЬ
    Видит обновленную таблицу с новой записью

ХАРАКТЕРИСТИКИ СИНХРОННОГО ВЗАИМОДЕЙСТВИЯ:

✓ Клиент отправляет запрос и ожидает ответа

✓ Предсказуемое поведение
✓ Легкая обработка ошибок

ИСПОЛЬЗОВАНИЕ:

СЛАЙД 14: СИНХРОННОЕ VS АСИНХРОННОЕ ВЗАИМОДЕЙСТВИЕ
================================================================================

СИНХРОННОЕ VS АСИНХРОННОЕ ВЗАИМОДЕЙСТВИЕ

СИНХРОННОЕ (HTTP REST) - ИСПОЛЬЗУЕТСЯ В ПРОЕКТЕ:

Преимущества:
✓ Простота реализации и понимания
✓ Стандартизированный подход (REST)
✓ Хорошая поддержка инструментами (Swagger, Postman)
✓ Легкая отладка и логирование
✓ Подходит для большинства веб-приложений
✓ Кеширование на уровне HTTP
✓ Stateless (без состояния) - легко масштабировать

Недостатки:
✗ Клиент должен ждать ответа
✗ Нет встроенной поддержки real-time обновлений
✗ Для длительных операций требуется polling

АСИНХРОННОЕ (WebSocket, Server-Sent Events, Message Queues):

Преимущества:
✓ Real-time обновления
✓ Двусторонняя связь
✓ Эффективно для чатов, уведомлений
✓ Меньше overhead для частых обновлений

Недостатки:
✗ Сложнее в реализации
✗ Требует постоянного соединения
✗ Сложнее масштабировать
✗ Больше ресурсов на сервере

ПОЧЕМУ ВЫБРАН HTTP:

1. Соответствие требованиям проекта:
   • CRUD операции не требуют real-time
   • Стандартные веб-запросы достаточны

2. Простота разработки:
   • Меньше сложности в коде
   • Легче тестировать

3. Совместимость:
   • Работает через любой прокси/файрвол
   • Стандартные инструменты разработки

4. Масштабируемость:
   • Stateless архитектура
   • Легко добавить балансировщик нагрузки

АЛЬТЕРНАТИВЫ HTTP (не использовались, но существуют):

• WebSocket - для real-time обновлений (чат, уведомления)
• gRPC - для микросервисов, бинарный протокол
• GraphQL - альтернатива REST, гибкие запросы
• Server-Sent Events (SSE) - односторонний поток от сервера
